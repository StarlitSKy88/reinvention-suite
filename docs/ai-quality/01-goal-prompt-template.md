# /goal 提示词模板

> **目的**：让 AI 持续工作到真正完成，避免"看起来完成"假象
> **适用于**：所有 30 分钟以上的复杂任务

---

## 🎯 5 要素模板

```markdown
# /goal [任务名称]

## 1. 结果标准（Result Criteria）
[可量化的成功标准，每个标准都可以被验证]

例如：
- 单元测试覆盖率 ≥ 80%
- 端到端流程在 5 步内可走通
- API 真实返回 200 + JSON（不是 mock）
- 浏览器截图显示 3 个核心元素
- 代码无 TypeScript 错误

## 2. 验证方式（Verification）
[用什么工具/方法验证标准已达成]

例如：
- `pnpm test` → 测试通过
- `curl http://localhost:3000/api/x` → 200
- `npx tsc --noEmit` → 0 errors
- `playwright screenshot` → 截图可见核心元素
- 手动点击走完整个流程

## 3. 禁区边界（Constraints）
[绝对不能动的东西]

例如：
- 不要修改已通过的测试
- 不要删除现有的种子数据
- 不要注释掉 TypeScript 错误
- 不要安装未经验证的新依赖
- 不要修改数据库 schema（除非明确要求）

## 4. 迭代记录要求（Iteration Log）
[每次尝试都要记录]

格式：
- 第 N 次尝试：什么改动了 → 效果如何
- 失败也要记录（不是只报成功）
- 累计用时

## 5. 卡住时的汇报规则（Blocker Protocol）
[3 次失败后必须停下来]

格式：
- 已尝试：X、Y、Z
- 每种尝试的效果：A、B、C
- 当前状态：仍然卡在 [具体问题]
- 建议尝试方向：P、Q、R
- 需要的输入：[用户决策点]
```

## 📋 我们项目的 /goal 实例

### 示例 1：完善简历匹配功能

```markdown
# /goal 把 match/jobs 改成真实匹配

## 1. 结果标准
- [ ] 用户上传简历后，能看到基于其技能的真实 Top 5 匹配
- [ ] 每个匹配的匹配分是真实计算的（不是写死的 87/78/72）
- [ ] 已匹配/待补全关键词是真实从用户技能 vs JD 关键词算出的
- [ ] 移动端显示正常（375x812）
- [ ] TypeScript 严格模式 0 错误
- [ ] next build 成功
- [ ] 真实调用 match engine（不是 mock）

## 2. 验证方式
- npx tsc --noEmit → 0 errors
- next build → 成功
- Playwright 截图 → 显示真实匹配分（不重复）
- 手动上传 mock PDF → 看到基于技能的不同匹配分
- curl /api/match → 真实返回

## 3. 禁区边界
- 不要修改 lib/match/engine.ts 的算法
- 不要修改其他已通过的页面
- 不要装新依赖
- 不要删除现有的 MOCK_JOBS（保留作为 fallback）

## 4. 迭代记录
- 第 1 次：尝试用 IndexedDB 读取简历 → 失败，IndexdDB 还没数据
- 第 2 次：fallback 到 MOCK_JOBS → 成功显示，但匹配分是固定的
- 第 3 次：用真实匹配引擎计算 → 用户看到不同的匹配分
- 第 4 次：考虑技能差异，让 mock 也变体
- ...

## 5. 卡住时的汇报
3 次失败后：
- 已尝试：A、B、C
- 当前：用户看到 5 个固定分数的 mock 岗位
- 建议：增加 MOCK_JOBS 变体，或暂时保留并标记 mock
```

## 🎯 通用 /goal 启动器

```bash
# 在 Claude Code 中使用
/goal [粘贴上述 5 要素]

# 在其他平台
@ai /goal [粘贴上述 5 要素]
```

## ⚠️ AI 偷懒的 5 个常见信号

| 信号 | 检测方法 | 应对 |
|---|---|---|
| 只写代码不测试 | 看到 `// TODO: test` 标记 | 要求先写测试 |
| 用 mock 假装工作 | 看到 `MOCK_DATA` 替代真实数据 | 要求去掉 mock 或标注 "MOCK" |
| 跳过错误处理 | 看到裸的 `try { }` 没有 catch | 要求加错误处理 |
| 编译过就当完成 | 跑 build 成功就交差 | 要求实际功能验证 |
| 文档与代码不一致 | 文档说实现了 X，代码找不到 | 要求同步更新 |
