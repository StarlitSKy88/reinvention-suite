# 6 步 SOP：把模糊品味转化为 AI 可执行标准

> **原始方法论**：看 baseline → 记录皱眉 → 归类维度 → 补充案例 → 多样性 → 迭代
> **应用目标**：让 AI 产出从"能用"提升到"好用"

---

## 📋 6 步 SOP

### ① 看 AI baseline 产出

**做什么**：
- 让 AI 用默认方式写一版
- 不给任何额外要求
- 完整跑一遍流程，看实际产出

**为什么**：
- 默认产出就是"平均水平"
- 知道平均水平在哪，才能知道提升空间

**应用到我们项目**：
```bash
# 让 AI 写一个页面
> 写一个 /test 页面，用默认方式

# 截图看效果
playwright screenshot
```

### ② 记录"皱眉点"

**做什么**：
- 把所有"看起来怪怪的"地方记下来
- 包括：命名、结构、错误处理、视觉、交互
- 不修改，只记录

**记录格式**：
```markdown
# /test 页面皱眉点

1. 加载状态用 "Loading..." 没有中文
2. 错误信息直接显示英文
3. 按钮 hover 没有反馈
4. 表单提交后没有成功提示
5. 移动端字号太小
...
```

**应用到我们项目**：
- 截图后认真看每个细节
- 列出所有"不完美"的地方
- 包括文案、布局、交互、错误处理

### ③ 归类维度

**做什么**：
- 把皱眉点分类
- 找规律，哪些是"风格"问题，哪些是"质量"问题

**常见的维度**：
- **可读性**：命名、注释、结构
- **性能**：加载速度、API 调用次数
- **可访问性**：键盘导航、屏幕阅读器
- **国际化**：多语言支持
- **错误处理**：边界情况、友好提示
- **视觉设计**：间距、颜色、字体
- **交互反馈**：loading / success / error 状态

**应用到我们项目**：
```markdown
# 维度分类

可读性: 1, 2
错误处理: 2, 4
视觉设计: 5
交互反馈: 1, 3, 4
国际化: 1
```

### ④ 补充具体案例

**做什么**：
- 每个维度举 1-2 个反例
- 描述"什么样算差、什么样算好"

**示例**（错误处理维度）：
- **差**：`throw new Error("API failed")` — 用户看不懂
- **好**：`显示"网络错误，请稍后重试"按钮`

**示例**（加载状态维度）：
- **差**：只显示 "Loading..."，没有进度
- **好**："正在解析 PDF（20%）... 正在脱敏（50%）... 正在提取结构（80%）... 完成（100%）"

**应用到我们项目**：
- 对每个皱眉点写"差 vs 好"对比
- 形成可执行的 checklist

### ⑤ 加入多样性

**做什么**：
- 不只看 AI 的一种产出
- 尝试不同的 prompt、不同的实现方式
- 比较后选最好的

**应用**：
- 写 3 种版本的同一页面（极简 / 标准 / 增强）
- 让用户或团队投票选
- 把胜出的方案作为标准

### ⑥ 迭代优化

**做什么**：
- 把"维度"和"标准"做成**自动化检查工具**
- 每次提交都跑这些检查
- 不达标就不通过

**最终交付物**：
- 一份 checklist
- 一组 lint 规则
- 一个 CI pipeline

## 🎯 应用到我们项目的具体示例

### 维度 1：反幻觉改写质量

**皱眉点**：
1. 改写后的简历可能丢失关键信息
2. AI 可能"润色"过度，看起来不真实
3. 没有标注每句话的来源

**好 vs 差**：
```
差：AI 自动"润色"简历 → 看起来漂亮但不真实
好：每个 bullet 标注来源 fact_id → 用户可追溯
```

**自动化检查**：
```typescript
// 每条改写后必须包含 factSources
if (!result.factSources || result.factSources.length === 0) {
  throw new Error('改写必须标注事实来源')
}

// 不能编造事实
if (newBullets > oldBullets) {
  throw new Error('不能编造新内容')
}
```

### 维度 2：UI 加载状态

**皱眉点**：
1. 加载时只显示 "Loading..."
2. 不知道进度（要等多久）
3. 错误信息不友好

**好 vs 差**：
```
差：只显示"加载中"
好：5 步骤实时进度（解析 20% → 脱敏 40% → 提取 60% → 分析 80% → 保存 100%）
```

**自动化检查**：
- 任何超过 2 秒的异步操作必须有进度提示
- 错误必须有友好提示 + 重试按钮

### 维度 3：移动端布局

**皱眉点**：
1. 字号太小
2. 按钮点击区域不够大
3. 横屏滚动

**好 vs 差**：
```
差：移动端文字挤在一起
好：单列布局，大按钮，可读字号
```

**自动化检查**：
- Playwright 截图 375x812
- 检查没有横屏滚动
- 按钮区域 ≥ 44px

## 🔧 工具化我们的 SOP

### Step 1：建立 Checklist

```markdown
# /resume/upload 质量 Checklist

## 功能完整性
- [ ] 拖拽上传工作
- [ ] 点击上传工作
- [ ] 5 步骤进度显示
- [ ] 错误友好提示
- [ ] 完成后跳转到分析页

## 视觉质量
- [ ] 黑底白字红字
- [ ] 衬线字体（Display）
- [ ] 巨大 01 背景数字
- [ ] 居中布局
- [ ] 移动端 375x812 正常

## 数据真实
- [ ] 客户端 PDF 解析（pdfjs-dist）
- [ ] PII 脱敏（不编造）
- [ ] LLM 真实调用（不是 mock）
- [ ] 失败有错误提示（不是静默 fallback）
```

### Step 2：建立 Lint 规则

```typescript
// .eslintrc.js
rules: {
  // 反幻觉检查
  'no-silent-fallback': 'error',
  
  // UI 反馈检查
  'require-loading-state': 'warn',
  
  // 错误处理检查
  'no-empty-catch': 'error',
}
```

### Step 3：建立 CI 检查

```yaml
# .github/workflows/quality.yml
- name: Type check
  run: npx tsc --noEmit
  
- name: Check no mock data
  run: |
    if grep -r "MOCK_JOBS\|MOCK_DATA\|MOCK_USERS" --include="*.ts" --include="*.tsx" apps/; then
      echo "Found MOCK data in production code"
      exit 1
    fi
    
- name: Check no TODO
  run: |
    if grep -r "TODO: implement" apps/web; then
      echo "Found TODO in production code"
      exit 1
    fi
```
